SECTION 01

프로젝트 기획서 (Project Proposal)

1.1 프로젝트 개요

프로젝트 정체성
프로젝트명SA-Tracker
프로젝트 성격서든어택 전적 조회 및 매치 분석 웹 서비스 (OP.GG 스타일)
대상 사용자서든어택 플레이어, "모" 클랜원
핵심 가치닉네임 하나로 모든 전적을 한눈에
핵심 지향점
  • 구조가 분명해야 한다 — Controller → Service → DTO 계층 준수
  • 나중에 다시 뜯어고치지 않아야 한다 — 최종 명세서 기반 구현
  • 기능이 늘어나도 폴더와 역할이 흔들리지 않아야 한다
  • 직관적인 UI를 통한 경기 기록 제공, 데이터 캐싱을 통한 성능 최적화

1.3 서비스 범위

1차 완성 범위 (MVP)

A
메인 페이지
B
플레이어 전적
C
매치 상세
D
클랜원 관리
A. 메인 페이지 (index.html)
  • 사이트 첫 진입 시 인트로 로딩 영상(MP4) 재생
  • 중앙 닉네임 검색창 제공 (엔터 또는 버튼 클릭 시 전적 조회)
  • "모" 클랜원 카드 목록 그리드 배치
  • 클랜원 클릭 시 해당 유저 전적 페이지로 즉시 이동
B. 플레이어 전적 페이지 (player.html)
  • 프로필 영역: 닉네임, 계급 이미지, 시즌 계급, 티어, 매너등급, 클랜명, 칭호
  • 전적 요약: 최근 승률, 최근 K/D, 무기군별(돌격/저격/특수) 최근 성과
  • 최근 매치 목록: 승리(녹색바) / 패배(빨간바) 구분, match_mode / match_type 필터
C. 매치 상세 페이지 (match-detail.html)
  • 매치 기본 정보: 맵, 모드, 매치 유형, 날짜(KST 변환)
  • 팀별 인원 목록: 유저별 킬 / 데스 / 헤드샷 / 데미지 / 어시스트
  • 두 팀 비교 테이블 형태
D. 클랜원 관리
  • MySQL clan_members 테이블에 클랜원 닉네임 저장
  • ouid는 최초 API 호출 시 DB에 캐싱하여 반복 호출 방지
  • 메인 페이지에서 DB 기반 렌더링

2차 확장 범위 (Post-MVP)

  • 전적 그래프 (승률/K/D 추이 시각화)
  • 최근 경기 성향 분석
  • 유저별 플레이 패턴 분석
  • 클랜 통합 대시보드
  • AI 기반 경기 분석 봇
SECTION 02

요구사항 정의서 (Requirements Definition)

2.1 기능적 요구사항

ID요구사항우선순위상세 설명
FR-001 로딩 인트로 재생 필수 사이트 첫 접속 시 MP4 로딩 영상 재생 후 fade-out → 메인 페이지 fade-in 전환. sessionStorage로 재방문 시 스킵 처리.
FR-002 닉네임 전적 검색 필수 사용자가 닉네임을 입력하면 NEXON API를 통해 ouid를 획득하고, 해당 유저의 전적 페이지로 이동.
FR-003 ouid DB 캐싱 필수 최초 조회한 ouid를 clan_members 테이블에 저장. 동일 닉네임 재조회 시 API 호출 생략하여 429 에러 방지.
FR-004 통합 기록 노출 필수 계급, 시즌 계급, 티어, 최근 승률, K/D, 무기군별 성과 등을 시각적으로 출력.
FR-005 매치 필터링 필수 match_mode(개인전/데스매치/폭파미션/진짜를모아라)와 match_type(일반전/클랜전/랭크전 등)에 따른 필터 제공.
FR-006 매치 상세 분석 필수 특정 매치 클릭 시 팀별 구성원과 유저별 상세 기록(킬, 데스, 헤드샷, 데미지, 어시스트) 비교.
FR-007 클랜원 카드 목록 필수 메인 페이지 하단에 DB 저장된 "모" 클랜원 닉네임을 카드형으로 배치. 클릭 시 전적 페이지 이동.
FR-008 메타데이터 연동 권장 계급/시즌계급/티어의 이미지 URL을 메타데이터 API에서 가져와 프로필에 표시.

2.2 비기능적 요구사항

ID요구사항상세 설명
NFR-001 보안 NEXON API KEY는 서버 사이드(application.properties)에서만 관리. 프론트엔드 노출 절대 금지. Spring Security 미사용 (전체 permitAll).
NFR-002 디자인 일관성 완전 블랙 배경(#0D0D0D) + 네온 그린(#7ED957) 포인트 컬러. 밝은 배경 절대 사용 금지. PPTX 디자인 톤 준수.
NFR-003 반응형 대응 데스크톱뿐만 아니라 모바일에서도 레이아웃 유지. 필요 시 mobile.css 별도 운영.
NFR-004 API 호출 최적화 ouid DB 캐싱을 적극 활용하여 429 Too Many Requests 에러 방지. 불필요한 중복 호출 제거.
NFR-005 시간대 변환 API 응답의 date_match(UTC)를 KST(+9시간)로 변환하여 표시.
NFR-006 한글 인코딩 닉네임, match_mode 등 한글 파라미터는 URLEncoder.encode(value, StandardCharsets.UTF_8) 필수 적용.

2.3 제약사항

  • 데이터 제공 시점: 게임 종료 후 평균 10분 소요
  • 조회 가능 기간: 2025년 1월 24일 이후 데이터만 조회 가능
  • ouid 변경 가능성: 게임 콘텐츠 변경 시 ouid가 변경될 수 있음 → 주기적 갱신 고려
  • 매치 목록: 최대 1,000개까지 조회 가능
  • match_result 규칙: "1"=승리, "2"=패배, "3"=무승부 (문자열 타입)
  • 진짜를 모아라 모드: 승패 미기록
  • 개인전/진짜를 모아라: 킬 수 높은 순 → 동일 시 데스 적은 순으로 순위 결정
  • clan_name 노출: 클랜전/퀵매치 클랜전/클랜 랭크전에서만 노출
SECTION 03

테이블 명세서 (Database Schema)

3.1 clan_members 테이블

클랜원 정보와 ouid 캐시를 저장하는 핵심 테이블. 1차 완성 범위에서는 이 단일 테이블로 운영한다.

컬럼명타입제약사항기본값설명
id BIGINT PK, AUTO_INCREMENT — 고유 식별자
user_name VARCHAR(50) NOT NULL, UNIQUE — 서든어택 닉네임
clan_name VARCHAR(50) — '모' 소속 클랜명
ouid VARCHAR(200) NULLABLE NULL NEXON API용 계정 식별자 (캐시)
created_at DATETIME — CURRENT_TIMESTAMP 레코드 생성 일시
updated_at DATETIME — CURRENT_TIMESTAMP ON UPDATE 레코드 수정 일시 (ouid 갱신 시 자동 업데이트)

3.2 DDL (Data Definition Language)

-- 데이터베이스 생성
CREATE DATABASE sa_match
  DEFAULT CHARACTER SET utf8mb4
  DEFAULT COLLATE utf8mb4_general_ci;

USE sa_match;

-- clan_members 테이블 생성
CREATE TABLE clan_members (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_name   VARCHAR(50)  NOT NULL COMMENT '서든어택 닉네임',
    clan_name   VARCHAR(50)  DEFAULT '모' COMMENT '클랜명',
    ouid        VARCHAR(200) COMMENT 'NEXON API ouid (캐시용)',
    created_at  DATETIME     DEFAULT CURRENT_TIMESTAMP,
    updated_at  DATETIME     DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_user_name (user_name)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 초기 클랜원 데이터 INSERT (예시)
INSERT INTO clan_members (user_name, clan_name) VALUES
('닉네임1', '모'),
('닉네임2', '모'),
('닉네임3', '모');

3.3 ERD (Entity Relationship Diagram)

1차 완성 범위 ERD

1차 범위에서는 clan_members 단일 테이블로 운영. 외부 API 데이터(유저 정보, 매치 기록 등)는 DB에 저장하지 않고 API 실시간 호출 → DTO 매핑으로 처리한다.

┌─────────────────────────────────────────┐
│              clan_members               │
├─────────────────────────────────────────┤
│  PK  id          BIGINT AUTO_INCREMENT  │
│  UK  user_name   VARCHAR(50) NOT NULL   │
│      clan_name   VARCHAR(50) DEFAULT '모'│
│      ouid        VARCHAR(200) NULLABLE  │
│      created_at  DATETIME               │
│      updated_at  DATETIME               │
└─────────────────────────────────────────┘
         │
         │  ouid 캐싱
         ▼
   ┌──────────────┐
   │ NEXON Open   │
   │ API (외부)    │  ← 실시간 REST 호출
   └──────────────┘

※ 2차 확장 시 매치 기록 캐싱 테이블, 유저 통계 테이블 등 추가 검토

개발 일정표 (Development Timeline)

PHASE 1 — 환경 구축 (1~2일)

PHASE 2 — 백엔드 코어 (3~4일)
API 연동 및 서비스 계층 구축
NexonApiConfig (RestTemplate Bean + API KEY) → NexonApiService (ouid 조회부터 단계별 테스트) → ClanMember Entity + Repository + ClanService (ouid 캐싱) → PlayerService / MatchService (데이터 조합) → MetaService (메타데이터 캐싱) → 각 API 호출 단위 테스트
PHASE 3 — 컨트롤러 + 기본 화면 (2~3일)
페이지 연결 및 기본 UI
HomeController (메인) + PlayerController (전적) + MatchController (매치 상세) 작성 → Thymeleaf fragments (header, footer) → index.html / player.html / match-detail.html 기본 구조 작성 → 데이터 바인딩 검증
PHASE 4 — 디자인 적용 (2~3일)
PPTX 톤 맞춤 UI 완성
common.css (다크 테마 변수, 리셋) → 페이지별 CSS 분리 적용 → 매치 카드 승패 컬러 바, glow 효과, 장식 블록 등 디자인 디테일 → 로딩 인트로 MP4 구현 → 반응형 대응
PHASE 5 — 필터 + 동적 기능 (1~2일)
검색/필터/인터랙션
검색창 엔터/버튼 동작 → match_mode / match_type 필터 셀렉트 박스 → 클랜원 카드 클릭 이동 → date_match UTC→KST 변환 표시 → 에러 페이지 처리
