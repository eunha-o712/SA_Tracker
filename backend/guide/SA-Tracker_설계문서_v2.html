1.2 아키텍처 개요

프론트엔드(React)와 백엔드(Spring Boot)를 완전히 분리합니다. 각각 별도 포트에서 실행되며 JSON으로 통신합니다.

FRONTEND
React + Vite
localhost:5173
📄 페이지 (Home, Player, Match)
🧩 컴포넌트 (SearchBar, MatchCard...)
📡 API 호출 (axios → 백엔드)
🎨 CSS 스타일링
⇄
BACKEND
Spring Boot
localhost:8085
🌐 REST API (/api/...)
🔑 NEXON API 호출 (API KEY 보관)
💾 MySQL (ouid 캐싱)
📦 DTO 변환 + 조합
왜 분리하는가? (초보자를 위한 설명)
  • Thymeleaf 방식 (v1): 서버가 HTML을 만들어서 보내줌 → 페이지 이동마다 새로고침 → 느림
  • React 방식 (v2): 브라우저가 직접 화면을 그림 → 데이터만 서버에서 받아옴 → 빠르고 부드러움
  • 비유: Thymeleaf = 식당에서 다 만들어서 가져다줌 / React = 재료만 받아서 내가 예쁘게 플레이팅

4.3 백엔드 REST API 엔드포인트

React 프론트엔드가 호출하는 백엔드 API 목록입니다. 모든 경로는 /api로 시작합니다.

메서드엔드포인트파라미터응답설명
GET /api/player name (닉네임) PlayerPageDto (JSON) 닉네임으로 기본정보+계급+티어+최근동향+매치목록 통합 조회
GET /api/player/matches name, mode(선택), type(선택) List<MatchDto> 매치 목록 필터 조회 (모드/유형 필터 적용)
GET /api/match/{matchId} — (경로변수) MatchDetailDto 매치 상세 기록 조회 (팀별 유저 목록 포함)
GET /api/clan/members — List<ClanMemberDto> DB에서 클랜원 전체 목록 조회
GET /api/meta/grades — List<GradeMetaDto> 통합 계급 메타데이터 (이미지 URL 포함)
GET /api/meta/tiers — List<TierMetaDto> 티어 메타데이터 (이미지 URL 포함)

WebConfig.java — CORS 설정 (필수)

@Configuration
public class WebConfig implements WebMvcConfigurer {
    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")          // /api로 시작하는 모든 경로
                .allowedOrigins("http://localhost:5173") // React 개발 서버
                .allowedMethods("GET")                // GET만 허용 (읽기 전용)
                .allowedHeaders("*");
    }
}

Controller 예시 — PlayerApiController

@RestController                           // ★ @Controller가 아닌 @RestController
@RequestMapping("/api")
public class PlayerApiController {

    private final PlayerService playerService;

    @GetMapping("/player")
    public PlayerPageDto getPlayer(@RequestParam String name) {
        return playerService.getPlayerPage(name);
        // ★ 뷰 이름이 아닌 DTO 객체를 반환 → JSON으로 자동 변환
    }
}

5.2 주요 파일 역할

App.jsx — 라우팅 설정

import { BrowserRouter, Routes, Route } from 'react-router-dom';
import HomePage from './pages/HomePage';
import PlayerPage from './pages/PlayerPage';
import MatchDetailPage from './pages/MatchDetailPage';

function App() {
  return (
    <BrowserRouter>
      <Routes>
        <Route path="/" element={<HomePage />} />
        <Route path="/player/:name" element={<PlayerPage />} />
        <Route path="/match/:matchId" element={<MatchDetailPage />} />
      </Routes>
    </BrowserRouter>
  );
}

api/api.js — API 호출 함수 모음

import axios from 'axios';

// axios 인스턴스 (백엔드 주소 고정)
const api = axios.create({
  baseURL: 'http://localhost:8085/api'
});

// 플레이어 전적 통합 조회
export const getPlayer = (name) =>
  api.get(`/player?name=${encodeURIComponent(name)}`);

// 매치 목록 (필터 포함)
export const getMatches = (name, mode, type) =>
  api.get('/player/matches', { params: { name, mode, type } });

// 매치 상세
export const getMatchDetail = (matchId) =>
  api.get(`/match/${matchId}`);

// 클랜원 목록
export const getClanMembers = () =>
  api.get('/clan/members');

SECTION 06

NEXON API 연동 명세서

외부 API 스펙은 v1과 완전히 동일합니다. React가 직접 NEXON API를 호출하지 않습니다 — 반드시 Spring Boot를 경유합니다 (API KEY 보안).

절대 금지

React 코드(.jsx, .js)에서 NEXON API를 직접 호출하면 안 됩니다. API KEY가 브라우저에 노출됩니다.

// ❌ 절대 하면 안 되는 코드
axios.get('https://open.api.nexon.com/suddenattack/v1/id', {
  headers: { 'x-nxopen-api-key': '내_API_KEY' }  // ← 브라우저에서 누구나 볼 수 있음!
});

// ✅ 올바른 방법: Spring Boot 백엔드를 경유
axios.get('http://localhost:8085/api/player?name=닉네임');
// → Spring Boot가 내부에서 NEXON API 호출 (KEY 안전)

SECTION 08

개발 일정표

PHASE 1 — 환경 구축 (1~2일)
프로젝트 초기 설정 (양쪽 다)
백엔드 STS에서 Spring Boot 프로젝트 생성 (Gradle, Java 21) → MySQL DB + 테이블 생성 → application.properties
프론트 npm create vite@latest → react-router-dom, axios 설치 → 기본 파일 구조 정리
PHASE 2 — 백엔드 REST API (3~4일)
Spring Boot API 서버 완성
백엔드 NexonApiConfig + WebConfig(CORS) → NexonApiService → ClanService (ouid 캐싱) → PlayerService / MatchService → @RestController 3개 작성 → Postman으로 JSON 응답 테스트
PHASE 3 — React 기본 페이지 (3~4일)
React 페이지 + 컴포넌트 골격
프론트 App.jsx 라우팅 설정 → api.js (axios 함수) → HomePage / PlayerPage / MatchDetailPage 기본 구조 → 백엔드 API 연결 테스트 → 데이터 바인딩 확인
PHASE 4 — 컴포넌트 세분화 (2~3일)
재사용 컴포넌트 + 상태 관리
프론트 SearchBar, ClanMemberCard, PlayerProfile, StatsSummary, MatchCard, MatchFilter, TeamTable 등 개별 컴포넌트 분리 → props 전달 + useState/useEffect 활용
PHASE 5 — 디자인 적용 (2~3일)
다크 테마 + 인트로 + 반응형
프론트 global.css (변수, 리셋) → 페이지별 CSS 분리 → 매치 카드 컬러바, glow 효과, 장식 블록 → LoadingIntro 컴포넌트 (MP4) → 반응형 미디어쿼리
PHASE 6 — 검증 + 마무리 (1~2일)
통합 테스트 + 에러 처리
전체 흐름 테스트 (검색→전적→매치상세) → API 에러 처리 (로딩 스피너, 404 페이지) → 크로스 브라우저 → 최종 점검
